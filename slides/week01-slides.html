Course Philosophy

Why AI-Assisted Development?

The World Has Changed 🌍

Before (2022)

  • Memorize syntax
  • Write every line by hand
  • Google + Stack Overflow
  • Copy-paste-adapt

Now (2025)

  • Understand concepts
  • Direct AI to generate code
  • AI + documentation
  • Prompt-generate-evaluate
The fundamental skill set for developers has shifted. You still need to understand what code does, but HOW you produce it has changed dramatically. This course embraces that shift rather than pretending it hasn't happened.

🤝 AI as Your Pair Programmer

"AI doesn't replace the developer — it replaces the blank page."
  • You define the architecture and requirements
  • AI generates boilerplate and suggests implementations
  • You evaluate, debug, and make final decisions
  • AI helps you iterate faster
Think of AI as a very fast but sometimes unreliable junior developer. It can produce code quickly, but you are the senior engineer who reviews, understands, and takes responsibility for what ships.

You Still Need to Understand

⚠️ If you can't evaluate AI output, you can't use AI effectively.

  • AI generates plausible code — not always correct code
  • You must recognize bugs, security holes, and bad patterns
  • Understanding fundamentals makes you a better prompter
  • Debugging requires knowing how things actually work
This is the most important slide of the night. We are NOT teaching you to blindly accept AI output. We are teaching you to be an effective AI-augmented developer. That requires understanding HTML, CSS, JavaScript, HTTP, and system design.

Part 2

The AI-Assisted Development Workflow

The Core Loop 🔄

1. Prompt → 2. Generate → 3. Evaluate → 4. Debug → 5. Iterate

This cycle typically takes 2–5 minutes per feature, not hours.

This five-step loop is the heartbeat of AI-assisted development. You'll internalize this over the quarter. Each step requires different skills — prompting is about communication, evaluation is about engineering judgment.

Step 1: Prompt ✍️

Clearly describe what you want the AI to produce.

  • Be specific — language, framework, constraints
  • Provide context — what exists already, what it connects to
  • State the goal — not just what, but why

Create a responsive navigation bar using HTML and CSS.
It should have a logo on the left, 4 nav links on the right,
and collapse into a hamburger menu on mobile (< 768px).
Use flexbox, no JavaScript needed for now.
A good prompt is like a good user story. It includes context, constraints, and acceptance criteria. We'll spend a whole section on prompt engineering later tonight.

Step 2: Generate ⚡

Let the AI produce code using your tool of choice:

Inline (Copilot)

  • Tab-completion in editor
  • Context from open files
  • Fast for small pieces

Chat (Copilot / ChatGPT / Claude)

  • Conversational interaction
  • Longer code generation
  • Explanations included
You'll develop intuition for when to use inline completion vs. chat. Inline is great for filling in function bodies. Chat is better when you need a whole component or want to discuss architecture.

Step 3: Evaluate 🔍

The most critical skill in this course.

  • Does the code do what you asked?
  • Are there obvious bugs or edge cases?
  • Is it using best practices or outdated patterns?
  • Are there security vulnerabilities?
  • Is it readable and maintainable?

🚨 Never deploy AI-generated code you don't understand.

This is where your CS fundamentals pay off. You need to read the code, trace the logic, and identify problems. We'll practice this skill throughout the quarter.

Steps 4 & 5: Debug & Iterate 🐛🔁

  • Debug: Use browser dev tools, console.log, breakpoints
  • Iterate: Refine your prompt with what you learned

# Iteration example:
Round 1: "Create a login form"
  → Too generic, missing validation

Round 2: "Create a login form with email validation,
  password minimum 8 chars, and error messages
  displayed below each field in red"
  → Much better output
Debugging AI code is just like debugging any code — except you can also ask the AI to help you debug it. And iteration is key: your second prompt is almost always better than your first.

Part 5

Prompt Engineering Fundamentals

Getting better output from AI

The SPICE Framework 🌶️

LetterPrincipleExample
S Specific "Use flexbox" not "make it look nice"
P Provide context "This is a React 18 component in a Next.js app"
I Include constraints "No external dependencies, ES modules only"
C Clarify output format "Return only the function, no explanation"
E Examples / Edge cases "Handle empty arrays and null inputs"
SPICE is a mnemonic I use for teaching prompt engineering. Each letter represents a dimension you can improve in your prompts. We'll see examples of each.

❌ Bad Prompt vs. ✅ Good Prompt

❌ Bad: "Make me a form"

✅ Good: "Create an HTML registration form with the following fields: full name (text, required), email (email type, required), password (min 8 chars, required), and a 'Sign Up' submit button. Use semantic HTML5 form elements. Style it with CSS — centered on the page, max-width 400px, with a card-like shadow. Add client-side validation that displays error messages below each field in red. Use vanilla JavaScript, no frameworks."

The difference is night and day. The bad prompt gives the AI no constraints, so it guesses at everything. The good prompt specifies fields, validation rules, styling approach, and technology constraints. Better input = better output.

Context Is King 👑

Always tell the AI about your environment:


# Include in your prompts:

"I'm building a Node.js Express API (v4.18) with
MongoDB using Mongoose. The app uses ES modules
(import/export syntax). I need to create a new
route for user authentication that..."

# vs.

"Create a login endpoint"

Pro tip: In Copilot, open related files in your editor — Copilot reads your open tabs for context.

GitHub Copilot automatically uses your open files as context. But for ChatGPT and Claude, you need to provide context manually. The more specific your context, the more accurate the output.

Iterative Refinement 🔄

Your first prompt is rarely your best. Iterate:

Round 1: "Create a todo list app"
→ Gets basic structure, but no persistence
Round 2: "Add localStorage persistence so todos survive page refresh"
→ Better, but no delete or edit functionality
Round 3: "Add ability to delete todos and mark them complete with a strikethrough"
→ Feature-complete
Round 4: "Refactor to use event delegation and add CSS transitions for adding/removing"
→ Polished
Building iteratively is natural with AI. Each round adds a layer. This mirrors real development — start simple, add complexity. The AI maintains context within a conversation, so each round builds on the last.

Prompt Patterns for Code

  • Generate: "Create a [component] that [does X] using [technology]"
  • Explain: "Explain this code line by line: [paste code]"
  • Debug: "This code throws [error]. Here's the code: [paste]. Fix it."
  • Refactor: "Refactor this to use [pattern/approach]: [paste code]"
  • Convert: "Convert this from [X] to [Y]: [paste code]"
  • Test: "Write unit tests for this function using [framework]"
  • Review: "Review this code for bugs, security issues, and best practices"
These are your go-to prompt templates. Memorize these patterns and you'll cover 90% of your AI interactions. We'll practice all of these throughout the quarter.

Part 6

What AI Gets Right vs. Wrong

Developing your evaluation instincts

✅ What AI Is Great At

  • Boilerplate code — HTML templates, config files, setup
  • Common patterns — CRUD operations, form validation, API calls
  • Syntax translation — "How do I do X in JavaScript?"
  • Documentation — JSDoc comments, README files
  • CSS styling — layouts, animations, responsive design
  • Regex patterns — email validation, parsing
  • Standard algorithms — sorting, searching, traversal
AI excels at well-trodden paths. If thousands of developers have written similar code before, AI has seen it in training data and can reproduce high-quality versions.

⚠️ Where AI Struggles

  • Business logic — your specific domain rules
  • Security — may generate vulnerable code (SQL injection, XSS)
  • State management — complex app state, race conditions
  • Up-to-date APIs — may use deprecated methods
  • Architecture decisions — when to use what pattern
  • Performance optimization — generates working but not optimal code
  • Edge cases — often ignores error handling
These are the areas where YOUR engineering judgment matters most. AI doesn't know your business requirements, your threat model, or your performance constraints. This is where you earn your salary as a developer.

Real Example: Spot the Bug 🐛

AI-generated code for "fetch user data and display it":


// AI generated this — can you spot the issues?
async function loadUser(userId) {
  const response = await fetch(`/api/users/${userId}`);
  const user = await response.json();

  document.getElementById('name').innerText = user.name;
  document.getElementById('email').innerText = user.email;
  document.getElementById('bio').innerHTML = user.bio;
}

loadUser(window.location.search.split('=')[1]);

Issues:

  • No check for response.ok — fails silently on 404/500
  • .innerHTML with user data = XSS vulnerability
  • URL parsing is fragile — use URLSearchParams
  • No null checks — crashes if elements don't exist
  • No loading or error state for the user
Walk through each issue. The XSS vulnerability is the most critical — if user.bio contains a script tag, it will execute. This is a real security hole that AI generates frequently. The fix is to use textContent instead of innerHTML, or sanitize the input.

The Fixed Version ✅


async function loadUser(userId) {
  if (!userId) {
    showError('No user ID provided');
    return;
  }

  try {
    const response = await fetch(`/api/users/${encodeURIComponent(userId)}`);

    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }

    const user = await response.json();

    // textContent is safe from XSS (unlike innerHTML)
    document.getElementById('name').textContent = user.name ?? 'Unknown';
    document.getElementById('email').textContent = user.email ?? '';
    document.getElementById('bio').textContent = user.bio ?? '';
  } catch (error) {
    console.error('Failed to load user:', error);
    showError('Could not load user profile. Please try again.');
  }
}

const params = new URLSearchParams(window.location.search);
loadUser(params.get('id'));
Compare this with the previous version. Every issue is addressed: error handling, XSS prevention with textContent, proper URL parsing, null coalescing, and user-facing error messages. This is what evaluation and debugging look like.

AI Confidence ≠ Correctness

AI presents all code with equal confidence.
It won't tell you "I'm not sure about this part."

  • AI-generated code that looks right can have subtle bugs
  • AI may "hallucinate" API methods that don't exist
  • AI may mix patterns from different framework versions
  • Always run the code and test edge cases
This is a crucial mental model. Unlike a human colleague who might say "I think this is right but you should double-check," AI presents everything with the same tone of certainty. You must be the skeptic.

Part 8

Tips for Effective AI Prompting

Level up your AI collaboration

Tip 1: Start with Structure 🏗️

Ask for the skeleton first, then flesh it out.


# Step 1: Get the architecture
"Outline the file structure for a Node.js Express
REST API with user authentication and a MongoDB
database. Just the file/folder names, no code yet."

# Step 2: Generate file by file
"Now create the server.js entry point based on
that structure."

# Step 3: Fill in details
"Now create the User model in models/User.js
using Mongoose with email, password (hashed),
and createdAt fields."
Don't ask AI to generate an entire application at once. Break it into layers: structure first, then individual files, then details. This gives you more control and makes evaluation easier.

Tip 2: Show, Don't Just Tell 📋

Include examples of input/output in your prompts:


"Write a function called formatCurrency that:
- Takes a number and a locale string
- Returns a formatted currency string

Examples:
  formatCurrency(1234.5, 'en-US')  → '$1,234.50'
  formatCurrency(1234.5, 'de-DE')  → '1.234,50 €'
  formatCurrency(0, 'en-US')       → '$0.00'
  formatCurrency(-50, 'en-US')     → '-$50.00'"

Examples act as implicit test cases.

Examples are incredibly powerful. They remove ambiguity and give the AI concrete test cases to satisfy. If the generated code doesn't match your examples, you immediately know something is wrong.

Tip 3: Use AI to Learn 📚

AI is an excellent tutor — use it to understand code:

Ask for Explanations

  • "Explain this regex"
  • "What does this CSS selector do?"
  • "Walk me through this async code"
  • "Why is this approach better?"

Ask for Alternatives

  • "Show me 3 ways to do this"
  • "What are the tradeoffs?"
  • "How would a senior dev write this?"
  • "What's the modern approach?"
Don't just use AI to write code — use it to learn. Ask why. Ask for alternatives. Ask for tradeoffs. This is how you build the understanding you need to evaluate AI output effectively.

Tip 4: Leverage Copilot Inline ⚡

Copilot reads your code context. Help it help you:


// Write a clear function name and comment — Copilot does the rest

// Validate email format and return true/false
function isValidEmail(email) {
  // Copilot suggests the implementation here ↓
}

// Format a date as "Month Day, Year" (e.g., "January 15, 2025")
function formatDate(date) {
  // Copilot suggests the implementation here ↓
}

// Fetch user data from API and handle errors
async function getUser(id) {
  // Copilot suggests the implementation here ↓
}

Good naming + comments = better Copilot suggestions

The function name and comment act as a mini-prompt. Copilot uses them plus your surrounding code to generate implementations. Clear, descriptive names produce dramatically better suggestions.

Tip 5: Always Verify 🔬

The Verification Checklist:

  • ☐ Does it run without errors?
  • ☐ Does it do what I asked?
  • ☐ Have I tested edge cases (empty input, null, large data)?
  • ☐ Is user input sanitized? (XSS, injection)
  • ☐ Are there hardcoded values that should be configurable?
  • ☐ Does it handle network failures gracefully?
  • ☐ Is the code readable by a teammate?
Keep this checklist handy. Before you commit any AI-generated code, run through these items. Over time this becomes second nature. We'll refer back to this checklist throughout the quarter.

Tip 6: Know When NOT to Use AI 🚫

  • Security-critical code — always have a human review
  • Complex business logic — AI doesn't know your domain
  • When learning fundamentals — sometimes write it yourself
  • Configuration with secrets — never paste credentials into AI
  • Novel algorithms — AI excels at common patterns, not invention

🔐 Never paste API keys, passwords, or private data into ChatGPT or Claude.

This is about judgment. AI is a tool, not a replacement for thinking. Know its limits and respect them. Security is especially important — never send sensitive data to a cloud AI service.
